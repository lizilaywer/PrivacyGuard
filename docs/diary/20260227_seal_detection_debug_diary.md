## 第一次测试：复选框不见了

### 问题描述
打开应用，进入高级设置，想勾选"印章"选项，结果发现——复选框根本不显示！

### 排查过程
一开始以为是代码问题，但检查 main.py 里的 DEFAULT_RULES，明明有：
```python
"印章": "__SEAL_DETECTION__"
```

那为什么 UI 不显示呢？突然想起来——config.json！这个项目有个 ConfigManager，如果 config.json 里没有印章规则，它会覆盖 DEFAULT_RULES。

果然，打开 config.json，里面没有印章配置项。

### 解决方案
在 config.json 里添加印章规则：
```json
"印章": {
  "pattern": "__SEAL_DETECTION__",
  "enabled": false,
  "description": "使用 OpenCV 自动检测并脱敏红色印章区域"
}
```

重启应用，复选框出现了！第一个问题解决。

## 第二次测试：印章检测不到

### 问题描述
复选框有了，勾选后打开带红色印章的 PDF，执行智能脱敏——结果什么都没发生。终端只显示：
```
[OCR] 使用引擎: rapidocr
```

没有 `[Seal Detection]` 的任何输出，说明代码根本没执行。

### 排查过程
这就有意思了。代码明明写好了，为什么不执行？

加了调试输出，发现 seal_detection_enabled 可能是 False。但明明勾选了啊！

再仔细看代码，发现了一个严重的问题：

**印章检测代码只在 `else` 分支（图像 PDF）执行！**

看这段代码：
```python
if is_text_pdf:
    # 文本 PDF 处理...
else:
    # 图像 PDF 处理...
    # v37.5.0: 印章检测（如果启用）
    if self.seal_detection_enabled:
        seal_rects = self._detect_seals(scan_img, SCAN_SCALE)
```

问题来了：如果用户的 PDF 是"文本型 PDF"（有文本层的扫描件或原生 PDF），代码会走 `if` 分支，印章检测根本不会执行！

### 这个 Bug 为什么没被发现？

这个 Bug 很隐蔽，因为：
1. 开发时可能只测试了纯图像 PDF（扫描件没有 OCR 文本层）
2. 真实场景下，很多 PDF 都有文本层（尤其是用扫描仪扫描后自动 OCR 的）
3. 文本型 PDF 的判断条件是 `len(page_text.strip()) > 20`，只要有超过 20 个字符就会被认为是文本型

### 解决方案
在文本 PDF 分支也添加印章检测代码：
```python
if is_text_pdf:
    # 处理文本敏感信息...

    # v37.5.0: 文本 PDF 也要检测印章（印章检测基于图像，与文本类型无关）
    if self.seal_detection_enabled and "__SEAL_DETECTION__" in self.rules:
        pix = page.get_pixmap(matrix=fitz.Matrix(SCAN_SCALE, SCAN_SCALE))
        img_data = np.frombuffer(pix.tobytes("png"), dtype=np.uint8)
        img_np = cv2.imdecode(img_data, cv2.IMREAD_COLOR)
        seal_rects = self._detect_seals(img_np, SCAN_SCALE)
        rects.extend(seal_rects)
```

这个改动的关键在于：**印章检测是图像处理，与 PDF 是否有文本层无关**。无论是什么类型的 PDF，都需要渲染成图像才能检测红色印章。

## 技术反思

### 1. 为什么会有 is_text_pdf 这个分支？

这是性能优化的考虑：
- 文本型 PDF 可以直接用 `page.search_for()` 搜索文本，速度快
- 图像型 PDF 需要 OCR 识别，速度慢

但这个优化导致了印章检测的遗漏——因为印章检测需要图像处理，而文本分支没有渲染图像。

### 2. 印章检测为什么需要渲染图像？

因为 OpenCV 处理的是像素数据，不是 PDF 的文本或矢量图形。即使 PDF 有文本层，印章也是作为图像嵌入的，必须渲染成像素才能检测颜色。

### 3. 为什么要判断 `len(page_text.strip()) > 20`？

这是一个启发式判断：
- 如果 PDF 页面有大量可提取的文本，大概率是文本型 PDF
- 阈值 20 是经验值，避免把只有几个字的图片误判为文本型

但这个判断不完美——有些扫描件 OCR 后可能有少量文字，但仍需要图像处理才能检测印章。

## 结语

今天的调试其实是一个很典型的开发场景：代码写好了，功能看起来是对的，但因为一个分支条件没有覆盖到，导致功能完全不工作。

这种问题很难发现，因为：
1. 语法没有错误
2. 逻辑看起来是对的
3. 特定场景下（图像 PDF）确实能工作

但一旦遇到"不符合预期"的场景（文本型 PDF），功能就失效了。

解决问题的方法是：
1. 加调试输出，确认代码是否执行
2. 追踪代码路径，找出遗漏的分支
3. 理解为什么会有这个分支（性能优化？历史遗留？）
4. 在所有需要的分支都添加功能

写这篇日记，一方面是记录今天的调试过程，另一方面也是提醒未来的自己：**当你修改一个有多个分支的代码时，要确保新功能在所有分支都能执行**。
