到晚上这个时间点，项目的真实状态已经不只是“Word 多字段替换做完了”，而是一路继续演化成了一个可以稳定继续迭代、可以回滚、可以打包、也能把上下文交给下一次开发的状态。今天我把这些分散的成果重新捏成了一个清晰的版本基线：`v37.7.1`，版本标识是 `37.7.1 - Mixed PDF OCR Hotfix`。

## 今天做了什么

今天最早的一段工作，其实是在读文档和接上下文。我先把 `STATUS.md`、`DEV_LOG.md`、`CHANGELOG.md`、`rollback_journal.md`、`PROJECT_INDEX.md`、`CLAUDE.md`、`QUICK_START_FOR_CLAUDE_CODE.md` 这些关键文件重新读了一遍，确认当时的稳定基线是 `v37.7.0`，并且已经完成了 Word 多字段替换、批量替换、批量入口并入“打开/拖拽”、高级设置整合、右侧替换后预览融合、运行时整改、以及 compare 空白热修复。

接着我又处理了打包方案。那一轮里，我把 `packaging/` 下的 active macOS / Windows 打包链统一到了当前基线，修了 Windows 版本资源、安装器版本、脚本逻辑和 macOS 版本同步、DMG fallback、签名 / 公证参数化这些问题。之后还在当前机器上实际跑了一遍 macOS 打包，确认可以生成 `.app`、`.dmg` 和 `.sha256`，只是那次没有做 Developer ID 签名和公证。围绕这件事，我也把“不做签名时别人如何安装”“做签名是否需要 Apple Developer 付费”这些现实问题梳理清楚了。

然后我又做了一轮更偏工程质量的事情：审查整个项目有没有严重问题和值得优先优化的性能点。那次结论很明确，主要问题在四个方向：路径校验不安全、`privacyguard` 导入时 OCR 依赖过重、文本型 PDF 重复命中会重复加框、Word 预览每次整页重绘会拖慢长文档。我把这些收敛成了一份按优先级排序的整改计划，单独写进了 `docs/current/PRIORITY_REMEDIATION_PLAN.md`，并且约定下次只要直接说“按这个计划执行”，就可以不再翻聊天记录。

后来这个计划也真的执行了。我按优先级把运行时整改做完：统一路径校验、让 `privacyguard` 和 worker 改成懒导入、抽出文本 PDF 去重逻辑、把 Word 预览改成按 `data-key` 局部更新、顺手修了配置持久化和版本源统一。紧接着测试时又暴露出一个新问题：打开单个 Word 后，第一次点“智能脱敏”，右侧“替换后预览”会整块空白。我继续排查，最后确认根因是 compare 模式从空白页切换时，误判右侧面板已经加载过，只做了局部更新，没有先加载完整 HTML。这个问题后来通过面板 loaded-source 状态和 `should_reload_word_panel(...)` 修掉了。

再往后，用户测出一个非常关键的新问题：混合型 PDF，也就是同一页里上面有文字层，下面有嵌入图片或扫描区域时，只有上面的文字被脱敏，下面图片里的同一关键词完全没反应。这一段是今天很重要的一次认知升级。原来当前 PDF 扫描逻辑是“文本页 / 扫描页”二选一，只要页面里检测到文本层，就根本不会对图片区域做 OCR。也就是说，导出链并没有问题，真正漏的是“扫描阶段没有为图片区域产出 OCR 矩形”。这件事确认后，我没有再继续在 `main.py` 和 worker 里各修一份，而是抽成共享逻辑 `privacyguard/ocr/mixed_pdf.py`，专门做三件事：提取图片块区域、对图片块裁剪后做 OCR、再把局部 OCR 框偏移回 PDF 页面坐标。主运行链和模块化 worker 都切过去，混合页的文本层和图片层终于都能一起脱敏了。

而到了现在这一轮，我做的是版本和交付层的收口。我把版本正式升级为 `v37.7.1`，把 `version.txt`、`main.py`、Windows 安装器默认版本、Windows EXE 版本资源都同步到了这个补丁版本。然后我把所有当前生效文档统一了一遍：`README.md`、`PROJECT_INDEX.md`、`CLAUDE.md`、`STATUS.md`、`DEV_LOG.md`、`QUICK_START_FOR_CLAUDE_CODE.md`、`RECOVERY_GUIDE.md`、`PROJECT_STRUCTURE.md`、`PROJECT_SUMMARY.md`、`docs/packaging/*`、`packaging/*`。我还把今天整天的对话和修改脉络写成了这篇日记，留给下次继续开发时的自己。

## 今天来回对话里，我学到的几件事

第一，很多需求表面上是在改 UI，实际上是在改交互模型。

比如最开始围绕 Word 批量替换的那几轮反馈，用户不是单纯想“多一个按钮”或者“少一个按钮”，而是在持续逼近一个更轻的交互路径：批量功能不要额外暴露独立入口，而是自然融进“打开/拖拽”；替换规则不要再占右上角空间，而是收进“高级设置”；规则界面本身不要被改坏，只是入口位置要更合理。真正的问题不是控件摆哪里，而是用户脑中对工作流的预期到底是什么。

第二，软件里很多“明明有这个功能却没生效”的问题，本质上都不是配置错了，而是处理链路压根没走到那一步。

Word 右侧空白是这样，混合型 PDF 图片漏脱敏也是这样。前者不是高亮逻辑错了，而是右侧页面没加载；后者不是关键词没配置，而是图片区没有进入 OCR 扫描链。只要把问题定位成“链路没走通”，查起来就会快很多。

第三，`main.py` 和 `privacyguard/*` 的双轨逻辑不能再继续放任漂移。

今天多个问题都反复证明了这一点。无论是路径校验、OCR 逻辑、文本型 PDF 去重，还是现在的混合型 PDF OCR，只修一边，另一边早晚还会重新长出 bug。后来我越来越明确一个原则：同类逻辑必须尽量抽共享 helper，主程序和 worker 都复用同一份实现。

第四，备份、回滚和日志这套东西不是“流程感很强的装饰”，而是真的能保命。

今天每一轮改动前后，我都继续按 checkpoint 记。尤其是后来做文档同步和打包同步时，我能明显感觉到，有 `rollback_journal.md`、有 `restore_checkpoint.sh`、有清晰的 cp 记录，整个人的决策压力会小很多。不是因为更勇敢了，而是因为知道自己有后路。

第五，文档如果不随着真实代码状态一起收敛，会比没有文档更危险。

今天我扫到不少文件，名字看起来很“当前”，内容却还是旧版本、旧路径、旧启动命令、旧版本获取方式。比如有些指南还在从 `main.py` 里硬读 `VERSION = "..."`，但现在真实版本源早就换成 `version.txt` 了。还有一些当前说明里还是 `v36.x` 或 `v31.9` 的叙述。这样的文档如果不清掉，下次接手的人不是“少一点帮助”，而是会被直接带偏。

## 今天的结论

今天这轮做完以后，我认为项目的状态更清楚了：

- 功能上，Word 多字段替换、批量替换、预览融合、混合型 PDF OCR 都已经在主链上了。
- 工程上，运行时整改、测试补齐、回滚链路、版本源统一也都已经落地。
- 交付上，当前 active 文档和打包方案终于和代码站到了一条线上。

还有没做完的事，当然有。比如 Phase 2 的每文件单独规则映射还没开始，批量规则模板管理也还没做，Word / PDF 这两条链依旧有继续模块化的空间。但现在至少可以比较踏实地说，项目不是停留在“能跑”的阶段，而是进入了“能继续稳定往前推”的阶段。

如果下次我再回来，我希望自己不要再从零开始猜。先看 `STATUS.md`、`DEV_LOG.md`、`CHANGELOG.md`、`rollback_journal.md`、`CLAUDE.md`，再看这篇日记，应该就能知道今天到底发生了什么、为什么这么改、以及下一步该从哪里接上。
